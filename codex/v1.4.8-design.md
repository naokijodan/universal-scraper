# v1.4.8 設計書 — reviveOrphanedSending を failed(unknown) 固定 + clientItemId 布石

## 背景

v1.4.7 で「リトライ既定 0」にしたが、実機で **`reviveOrphanedSending` 経路（経路 A）** が発火して二重書き込みが継続発生。

実機ログ（2026-05-16 椛島さん採取）:
```
[queue] revived 10 orphaned sending items, skipped 20 in-flight
```

3者協議 (Claude + GPT-5 + Gemini) で以下が合意:
- **経路 A の即時止血が必須**
- 単純な `failed` ではなく **「失敗」と「不明 (要 Sheets 確認)」を区別すべき**（Gemini の Validation Fatigue 警告）
- **payload に `clientItemId` を含める布石を v1.4.8 で先行**（GAS 側の v2.0 idempotency 実装への準備）

## ゴール

1. **`reviveOrphanedSending` を「sending → failed (unknown_remote_state)」固定に変更**（waiting に戻さない）
2. **payload に `clientItemId` を含める**（GAS 側コード変更不要、布石）
3. **UI で「失敗」と「不明（要 Sheets 確認）」を視覚的に区別**
4. バージョン v1.4.7 → **v1.4.8**
